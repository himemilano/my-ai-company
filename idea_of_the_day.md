# 「どっちもイイ。」Webアプリ実装計画書

## 1. はじめに

### 1.1. アプリ概要

本ドキュメントは、現代人の日常に潜む小さなストレスを解消する、ブラウザだけで動作するミニマルなWebアプリ「どっちもイイ。」の実装計画書である。
ユーザーが些細な選択に迷った際、アプリが瞬時に意思決定をアシストし、無駄な思考時間や決断疲れを軽減することを目的とする。

### 1.2. 目的

*   日常の些細な選択における決断疲れと、それによる思考時間の消費を解消する。
*   ユーザーがどれを選んでも大差ない選択において、迷うプロセスをスキップし、迅速に行動へ移れるよう支援する。
*   過去の選択をシンプルに振り返ることで、「なんだ、結局どうにかなるんだ」という安心感を提供する。

### 1.3. ターゲット層

日常の些細な選択に迷いがちで、その度に時間や精神力を消費してしまう現代人。特に、選択肢が多すぎて決断疲れを感じやすい20代後半〜40代のオフィスワーカーや主婦層。

## 2. 機能要件

### 2.1. 「今日の選択」クイック決断アシスト

#### 2.1.1. 機能概要

ユーザーが迷っている2つ以上の選択肢を簡潔なキーワードで入力し、「決めて！」ボタンをタップすると、アプリがランダムなアルゴリズムに基づき瞬時に1つの選択肢を提案する。提案に納得できない場合は、再度提案を求めることが可能。

#### 2.1.2. UI/UX 要件

*   **入力フォーム:**
    *   複数の選択肢を改行区切りで入力できるテキストエリアを設置。
    *   プレースホルダーの例: 「ランチ: ラーメン\nランチ: パスタ\nランチ: カレー」
*   **アクションボタン:**
    *   入力完了後、提案を促す「決めて！」ボタン。
    *   提案表示後、別の提案を求める「もう一度」ボタン。
    *   提案内容を確定し、「選択の足跡」に記録する「これで決定」ボタン。
*   **提案表示エリア:**
    *   アプリが選択した項目を大きく、明瞭に表示する。
    *   必要に応じて、短いアニメーション（例: カウントダウン、ルーレットのような表示）を挟むことで、決断をエンターテイメント化する。

#### 2.1.3. ロジック要件

*   **入力パース:** テキストエリアに入力された文字列を改行コードで分割し、選択肢の配列として処理する。空行や重複は無視する。
*   **ランダム選択アルゴリズム:**
    *   入力された選択肢の中から、純粋なランダムで1つを抽出する。
    *   （将来的な拡張として）特定の選択肢に重み付けを加えたり、短いカウントダウン後に選択する演出を実装することも検討可能。
*   **再提案:** 「もう一度」ボタンがタップされた場合、再度ランダム選択を実行し、結果を表示する。前回と同じ選択肢が提案される可能性もある。
*   **ログ連携:** 「これで決定」ボタンがタップされた際、選択された項目を「選択の足跡」機能へ渡す。

#### 2.1.4. データ要件

*   ユーザーが入力した選択肢データの一時的な保持（セッション内、またはクライアントサイドストレージに保持し、ブラウザを閉じても入力内容が残るようにする）。

### 2.2. 「ジャンル別」ランダム提案ジェネレーター

#### 2.2.1. 機能概要

ユーザーが漠然としたカテゴリ（例：「今日の気分」「夕食のメイン」）を入力するか、プリセットされたカテゴリを選択することで、アプリがそのカテゴリに関連するシンプルで汎用的な選択肢をランダムで提案する。ユーザーは選択肢を自由にカスタマイズ・追加・削除できる。

#### 2.2.2. UI/UX 要件

*   **ジャンル選択/入力:**
    *   プリセットされたジャンルを選択できるドロップダウンメニューまたはボタンリスト。
    *   ユーザーが新しいジャンル名を自由に入力できるテキストフィールド。
*   **選択肢カスタマイズ:**
    *   各ジャンルに対応する選択肢の一覧を表示するリスト。
    *   選択肢の追加、編集、削除ができるフォーム/ボタン。
    *   （例: 「今日の気分」ジャンルに「映画鑑賞」「読書」「散歩」「仮眠」などが表示され、ユーザーが「瞑想」を追加できる）
*   **アクションボタン:**
    *   ジャンル選択/入力後、提案を促す「提案！」ボタン。
*   **提案表示エリア:**
    *   アプリが選択した汎用的な項目を明瞭に表示する。

#### 2.2.3. ロジック要件

*   **ジャンル管理:** プリセットジャンルとユーザー定義ジャンルを管理する。
*   **選択肢リスト管理:**
    *   各ジャンルに紐づく選択肢のリストを管理する。
    *   ユーザーによる選択肢の追加、編集、削除機能を実装。
*   **ランダム提案:**
    *   選択されたジャンルに紐づく選択肢リストの中から、ランダムに1つを抽出する。

#### 2.2.4. データ要件

*   プリセットジャンルとその初期選択肢リスト。
*   ユーザーがカスタマイズしたジャンル、および各ジャンルに紐づく選択肢リストをクライアントサイドストレージ（LocalStorageなど）に永続化する。

### 2.3. 「選択の足跡」シンプルログ

#### 2.3.1. 機能概要

「今日の選択」機能でユーザーが決定した選択肢を、日付と結果のみを一行のテキストで記録し、タイムライン形式で表示する。詳細な内容や感情は記録せず、あくまで「何を選んだか」という結果のみを追うことで、記録の負担を最小限に抑える。

#### 2.3.2. UI/UX 要件

*   **ログ表示:**
    *   タイムライン形式で、新しいログが上に来るように表示されるリスト。
    *   各ログエントリは「YYYY/MM/DD [選択肢内容]」の形式で表示する。
    *   （例: `2023/10/27 ラーメンに決定`）
*   **操作性:**
    *   シンプルに表示するのみとし、各ログの詳細表示や編集機能は設けない（最小限の機能に留めるため）。
    *   必要に応じて、ログのクリアボタンを設けることを検討。

#### 2.3.3. ロジック要件

*   **ログ追加:** 「今日の選択」機能から渡された選択肢を、現在の日付とともにログデータに追加する。
*   **ログ表示:** 保存されたログデータを日付の降順（新しいものが上）で取得し、指定されたフォーマットで表示する。

#### 2.3.4. データ要件

*   ログデータ（`date: string`, `choice: string` のペア）をクライアントサイドストレージ（LocalStorage, IndexedDBなど）に永続化する。

## 3. 非機能要件

### 3.1. 動作環境

*   主要なWebブラウザ (Chrome, Firefox, Safari, Edge) の最新版で動作すること。
*   レスポンシブデザインに対応し、PC、タブレット、スマートフォンで適切に表示・操作できること。

### 3.2. パフォーマンス

*   クライアントサイドでの処理が中心であるため、高速なレスポンスタイムを維持し、ユーザーがストレスなく利用できること。
*   ページロード時間、機能実行時の遅延を最小限に抑えること。

### 3.3. アクセシビリティ

*   （フェーズ1では必須としないが、将来的に考慮）WCAGに準拠した基本的なアクセシビリティを確保する。

### 3.4. データ永続性

*   ユーザーが入力・設定したデータ（ジャンル、選択肢、ログ）は、ブラウザのローカルストレージに永続化され、次回アクセス時にもデータが保持されること。

## 4. 技術スタック（想定）

*   **フロントエンド:** HTML5, CSS3, JavaScript (Vanilla JS または軽量なフレームワーク/ライブラリ Vue.js / React.js)
*   **データ永続化:** Web Storage API (LocalStorage) または IndexedDB
