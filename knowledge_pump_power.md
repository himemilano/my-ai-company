3. 実装のための厳密な換算ロジック（AI用）
AIがプログラミング（Dart/Python等）で計算機を実装する場合、以下のステップを順守する必要があります。
Step 1: 圧力単位の正規化
試験問題では圧力が MPa で与えられることが多いため、公式の単位である kgf/cm 
2
  に変換します。
ロジック: P[kgf/cm 
2
 ]=P[MPa]×10.197
試験上の簡略化: 計算問題では 1MPa≈10kgf/cm 
2
  として計算させるケースが一般的です。
Step 2: 効率のデシマル化
％で与えられた効率を計算可能な数値に直します。
ロジック: η=Efficiency [%]/100
Step 3: 原動機補正（内燃機関の場合）
原動機がエンジン（内燃機関）の場合、電動機よりも出力のロスが大きいため、2割増しの出力を確保する必要があります。
補正ロジック: Required_Engine_Power=Calculated_Power×1.2
4. プログラミング用擬似コード (Pseudocode)
def calculate_pump_horsepower(pressure_mpa, flow_rate_lmin, efficiency_percent, is_engine=False):
    # 1. 単位換算: MPa -> kgf/cm2
    p_kgf = pressure_mpa * 10.197
    
    # 2. 効率の小数化
    eta = efficiency_percent / 100.0
    
    # 3. 馬力(PS)の基本計算
    h_ps = (p_kgf * flow_rate_lmin) / (450 * eta)
    
    # 4. エンジン補正 (エンジンなら2割増し)
    if is_engine:
        h_ps = h_ps * 1.2
        
    return h_ps
5. 試験対策：引っ掛けパターンと注意点
① 単位の罠：MPa vs kgf/cm 
2
 
パターン: 問題文に「吐出圧力 2.0 MPa」とあるのに、そのまま公式の P に「2.0」を代入してしまうミス。
対策: 公式の定数「450」や「612」は圧力が kgf/cm 
2
  であることを前提としているため、必ず10倍（正確には10.197倍） してから代入すること。
② 効率 η の代入ミス
パターン: 効率80%の場合に、分母に「80」と入れてしまう。
対策: 分母は必ず「0.8」などの小数にする。
③ ポンプ効率の推定（数値が与えられない場合）
パターン: 「泥水を使用する場合の所要馬力を求めよ」とあり、効率が明示されない。
対策: 資料に基づき、清水なら0.8〜0.9、泥水なら0.75〜0.85 の範囲の数値を想定する
。
④ エンジン（内燃機関）か電動機か
パターン: 計算結果に選択肢があるが、エンジン使用を前提とした「1.2倍」を忘れている。
対策: 問題文の主語が「エンジン馬力」を求めているのか「電動機出力」を求めているのかを真っ先に確認する。エンジンなら計算結果の最後に ×1.2 を行う
。
このロジックをアプリの計算シミュレーターに組み込むことで、ユーザーは現場の数値を入力するだけで、ポケットブック理論に基づいた正確な機材選定の回答を得ることが可能になります。
